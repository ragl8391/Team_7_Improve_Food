## Overview

This document describes the **database collections**, **document schemas**, and **data access methods** for the *Improve Food* application. It serves as a developer-facing design blueprint that explicitly defines how data is structured, stored, and validated within our MongoDB ecosystem.

This document answers the following design questions:
- What collections exist in the MongoDB database?
- What keys, data types, and constraints do those documents contain?
- What are the data access routines utilized by the Flask backend API?
- Which application pages and routes depend on which data elements?
- How do we test both the schema records and the API access routines?

The backend interfaces with **MongoDB** using **PyMongo** inside a lightweight Flask service architecture (`backend_api.py`).

# Database Collections

The application relies on three distinct collections within the `improve_food` database instance:
- `food_items`
- `users`
- `restaurants`

Each document structure is detailed down below.

---

## 1) Collection: food_items

### Collection Description
Stores the active food inventory documents, their metadata, reservation states, and delivery permissions.

### Fields
| Field Name | Data Type | Description | Constraints / Validation |
|------------|-----------|-------------|--------------------------|
| _id | ObjectId | Unique identifier generated by MongoDB | Primary Key |
| name | String | Public name of the food item | NOT NULL |
| quantity | String / Int| Total amount of items available | NOT NULL |
| expiry | String | Expiration date statement or date string | NOT NULL |
| status | String | Current transaction state of the item | CHECK (available, reserved) |
| delivery_available | Boolean | Flags if the item is eligible for delivery | Default: true |

### Collection Tests

**Use Case Name:** Create Food Item Document  
**Description:** Verify a new food surplus record can be correctly stored in the collection.  
**Pre-conditions:** MongoDB instance is running and reachable via PyMongo client.  
**Test Steps:**
1. Send a POST request containing a name, quantity, and expiry to `/add-item`.
2. Verify that the document inserts successfully with an `available` status string.  
**Expected Result:** Document is created with a clean `_id` and correct properties.  
**Actual Result:** Document successfully added; JSON response yields code 201.  
**Status:** Pass  
**Post-conditions:** Food item data is persistently stored inside the `food_items` collection.

## 2) Collection: users

### Collection Description
Manages system authentication profiles and physical coordinate points for location-distance computations.

### Fields
| Field Name | Data Type | Description | Constraints / Validation |
|------------|-----------|-------------|--------------------------|
| _id | ObjectId | Unique user identifier | Primary Key |
| email | String | Unique user account email address | NOT NULL, Must be Unique |
| name | String | User display moniker | NOT NULL |
| user_lat | Double / Float | Physical latitude value of user location | CHECK (-90.0 to 90.0) |
| user_lon | Double / Float | Physical longitude value of user location| CHECK (-180.0 to 180.0) |

# Data Access Methods

Every operational route inside `backend_api.py` interacts directly using these specific access operations.

---

## Access Method: add_item

### Description
Parses incoming multi-part form data to append an inventory item into `food_items` collection.

### Parameters
- `item_name` (string)
- `quantity` (string)
- `expiry` (string)

### Return Values
- JSON payload mirroring the saved data fields alongside its fresh string `_id` metadata.

### Tests

**Use Case Name:** Endpoint Item Insertion Validation  
**Pre-conditions:** PyMongo context initialized.  
**Test Steps:**
1. Invoke the API route matching complete form attributes.  
**Expected Result:** MongoDB updates with custom item status code 201.  
**Post-conditions:** A clean target record exists inside the target database collection.  

---

## Access Method: reserve_item

### Description
Performs an transactional state shift to lock down an inventory item if it falls within the required geographic boundaries.

### Parameters
- `item_id` (string)
- `user_lat` (string / float)
- `user_lon` (string / float)
- `restaurant_lat` (string / float)
- `restaurant_lon` (string / float)

### Return Values
- JSON string confirmation payload with relevant status codes (200 for pass, 400 for structural boundary errors, 409 for dynamic state collisions).

### Tests

**Use Case Name:** Guard Delivery Range Boundaries (Under 5 miles)  
**Pre-conditions:** Target document is tagged `available`.  
**Test Steps:**
1. Fire a POST request passing valid coordinate points corresponding to a distance under 5 miles.  
**Expected Result:** Haversine formula calculates sub-5-mile space, sets target element to `reserved`, and emits code 200.  
**Status:** Pass  

**Use Case Name:** Reject Excessive Delivery Range Boundaries (Over 5 miles)  
**Pre-conditions:** Document is tagged `available`.  
**Test Steps:**
1. Execute the endpoint route passing coordinates separated by a distance greater than 5 miles.  
**Expected Result:** System returns code 400 stating "Cannot deliver when distance is greater than 5 miles".  
**Status:** Pass
